Контрольні питання та відповіді
1. Основи Git
Яка різниця між Git та GitHub? Git — це програма (система контролю версій), яка працює локально на комп'ютері. GitHub — це веб-сервіс (хостинг), де зберігаються репозиторії Git в інтернеті.

Що означає "розподілена" система контролю версій? Це означає, що кожен розробник має на своєму комп'ютері повну копію всієї історії проєкту, а не лише поточні файли. Можна працювати без інтернету.

Яка різниця між git init та git clone? git init створює новий пустий репозиторій у поточній папці. git clone копіює вже існуючий репозиторій з віддаленого сервера.

2. Staging Area та Commits
Що таке Staging Area і навіщо вона потрібна? Це проміжна зона ("індекс"), куди ми додаємо файли перед комітом (git add). Вона дозволяє обрати, які саме зміни увійдуть у наступний коміт, а які — ні.

Яка різниця між git add . та git add -A? У сучасних версіях Git різниці майже немає — обидві команди додають усі зміни (нові, змінені та видалені файли) з поточної директорії.

Як написати гарне commit message? Використовувати теперішній час ("Add feature", а не "Added"), писати короткий заголовок (до 50 символів) і, за потреби, детальний опис через пустий рядок.

Як виправити останній коміт? Використати команду git commit --amend. Вона дозволяє змінити повідомлення або додати забуті файли до останнього коміту.

3. Гілки та злиття
Що таке гілка в Git і чому вони "легковажні"? Гілка — це просто рухомий вказівник на певний коміт. Вони "легковажні", бо створення гілки займає мінімум пам'яті (41 байт) і відбувається миттєво, незалежно від розміру проєкту.

Яка різниця між git merge та git rebase? merge об'єднує історії, створюючи новий коміт злиття (зберігає реальну хронологію). rebase переписує історію, накладаючи коміти один за одним, ніби робота йшла послідовно (робить історію лінійною).

Що таке fast-forward merge? Це тип злиття, коли Git просто пересуває вказівник гілки вперед, не створюючи додаткового "merge commit". Це можливо, якщо в основній гілці не було нових змін.

Коли виникають merge конфлікти і як їх уникнути? Виникають, коли в двох гілках змінено одні й ті самі рядки коду. Уникнути можна частими комітами, синхронізацією (git pull) та роботою над різними файлами.

Чому не можна робити rebase публічних комітів? Тому що rebase змінює хеші комітів (переписує історію). Якщо хтось інший вже скачав ці коміти, у нього виникнуть проблеми при синхронізації.

4. Віддалені репозиторії
Що таке origin? Це стандартна назва (псевдонім) для основного віддаленого репозиторію, з якого було клоновано проєкт.

Яка різниця між git fetch та git pull? fetch завантажує зміни з сервера, але не застосовує їх до вашого коду. pull робить fetch, а потім одразу намагається зробити merge у вашу гілку.

Що робить git push --force і чому це небезпечно? Вона примусово перезаписує історію на сервері вашою локальною історією. Це небезпечно, бо може видалити коміти інших розробників.

Як видалити віддалену гілку? Команда: git push origin --delete назва_гілки.

5. Скасування змін
Яка різниця між git reset --soft, --mixed та --hard?

--soft: скасовує коміт, але залишає зміни в Staging area (готові до коміту).

--mixed: скасовує коміт і Staging, зміни залишаються просто як незбережені файли.

--hard: повністю знищує всі зміни і повертає стан до коміту.

Яка різниця між git reset та git revert? reset видаляє історію (повертається назад), а revert створює новий коміт, який скасовує зміни попереднього (безпечно для історії).

Як скасувати git add (видалити файл зі staging)? Команда: git restore --staged файл (або git reset HEAD файл).

Як повернути видалений файл? Команда: git restore файл (або git checkout HEAD -- файл).

6. Безпека
Що не повинно потрапляти в Git репозиторій? Паролі, API-ключі, папка node_modules, файли конфігурації з секретами (.env), бінарні файли збірки.

Що робити якщо випадково закомітили пароль? Треба негайно змінити пароль у сервісі, де він використовується. Потім видалити його з історії Git (наприклад, через BFG Repo-Cleaner) або видалити файл і зробити новий коміт (але в історії він залишиться).
HTTPS vs SSH для GitHub — що краще? SSH краще, бо не треба постійно вводити пароль, і це безпечніше.

Що таке Personal Access Token і навіщо він потрібен? Це довгий код, який використовується замість пароля при роботі через HTTPS, оскільки GitHub заборонив використання звичайних паролів для командного рядка.

7. GitHub
Що таке Pull Request? Це запит на влиття змін з вашої гілки в основну гілку репозиторію. Він дозволяє обговорити код (Code Review) перед прийняттям.

Яка різниця між Issues та Pull Requests? Issues — це просто обговорення проблем або задач (текст). Pull Requests — це конкретні пропозиції змін у коді.

Що таке Fork і навіщо він потрібен? Це повна копія чужого репозиторію у вашому акаунті. Потрібен, щоб вносити зміни в чужі проєкти без прямого доступу до них.

Що таке GitHub Actions? Це інструмент для автоматизації (CI/CD): автоматичний запуск тестів, збірка проєкту або деплой при кожному пуші.

8. Best Practices
Як часто треба робити коміти? Часто. Кожен логічно завершений крок (навіть маленький) має бути окремим комітом ("атомарні коміти").

Скільки часу має жити feature branch? Якнайменше. 1-2 дні, максимум тиждень, щоб уникнути складних конфліктів при злитті.

Як організувати git workflow в команді? Використовувати гілки для нових функцій, робити Pull Requests, проводити Code Review перед злиттям у main.

Чи треба комітити node_modules або інші залежності? Ніколи. Вони мають бути в .gitignore. Залежності встановлюються кожним розробником окремо через npm install.
